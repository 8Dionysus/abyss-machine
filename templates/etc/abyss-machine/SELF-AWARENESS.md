# Abyss Machine Self-Awareness

## Role

`abyss-machine self-awareness` is the host-owned causal-temporal-spatial
understanding layer over read-only stack and machine evidence.

It answers what happened, when it happened, where it sits across host, process,
container, service, route, endpoint, owner surface, and context, what evidence
is connected, what remains missing, and which owner-gated reaction candidate is
appropriate.

The layer now treats the running stack and local AI routes as organs of the
same evidence loop. It does not stop at LGTM health: it inventories active
stack services, model routes, RAG/memory surfaces, nervous freshness gates,
resource/mode gates, investigation replay, and governed reactions/responses.

## Owner Boundary

- `abyss-stack` owns Prometheus, Grafana, Loki, Alloy, Alertmanager,
  Tempo-or-trace runtime, Postgres, Neo4j, `rag-api`, `route-api`,
  `langchain-api`, dashboards, scrape config, log shipping config, exporters,
  model roots promoted by the stack, and stack repositories.
- `abyss-machine` reads stack endpoints and writes only machine-owned contracts,
  readmodels, validators, change records, and decision rationale.
- `abyss-machine` owns host AI evidence for STT, embeddings, LLM profiles, TTS,
  NPU, warm-E2B/gemma4 resident status, and on-demand E4B/Qwen escalation
  gates. Stack runtime promotion remains stack-owned.
- Missing stack capabilities become owner-routed requirements, not host-layer
  runtime changes.
- Self-awareness never starts, stops, reloads, or reconfigures stack services.
- Alerts become reaction candidates only. Responses remain owner-gated.
- Trace IDs, request IDs, session IDs, goal IDs, task IDs, and run IDs must not
  become Loki labels. Keep them in structured log bodies, metadata, or machine
  readmodels.

## Coverage Planes

The current full-coverage route proves these planes independently before
claiming readiness:

- stack observability: Prometheus/PromQL, Grafana health, Loki/LogQL,
  Alloy/OTel, Alertmanager lifecycle, and optional trace backend requirement;
- stack closure dossier: one owner-facing packet that joins open stack-owned
  requirements, live probes, readiness, dependency order, runbooks, verifier
  commands, and no-stack-mutation policy;
- active stack services: `route-api`, `rag-api`, `langchain-api`, Postgres, and
  Neo4j presence from host/container and stack evidence;
- local AI: STT, embeddings, LLM text, TTS, NPU, resident warm-E2B with
  source-model, device, runtime, profile, eval, serving health, monitor,
  resource, candidate, and eval detail, and resource/mode-gated E4B/Qwen
  escalation routes;
- memory and space: RAG trace/eval/validation, generated graph/maps, nervous
  retrieval, bounded retrieval packets, freshness gates, spatial overlays, and
  explicit Postgres/Neo4j read-route requirements;
- bounded context packet: `self-awareness context` joins correlation contexts,
  memory-space, stack handoff, warm-E2B resident worker, governance gates, and
  escalation gate status into a compact read-only packet for resident/operator
  reasoning;
- signal fabric: every normalized observation event carries actor, entity,
  temporal, spatial, context-link, source-query, evidence-route, label-policy,
  and non-mutating access metadata;
- reasoning loop: bounded query, timeline, spatial graph, causal episodes,
  checkpointed investigation, replay, brief, cycle proof, export, and
  validation;
- portable export: manifest-indexed latest artifacts with expected schema,
  current schema, history path, size, hash, evidence ref, owner boundary, and
  non-mutating policy;
- host governance: memory class, resource plan/run/orchestrator state, mode
  effective state, launch limits, thermal posture, and no-stack-mutation policy
  before model or response escalation;
- failure coverage: missing/stale stack observability, trace/backend gaps,
  database/graph read-route gaps, LangGraph/API observability gaps,
  resource-denied work, stale nervous semantics, secret redaction, cardinality
  risk, and bounded stack downtime;
- governance: reaction candidates and response routes remain approval-gated,
  non-executing, and evidence-cited.

Freshness gates precede reasoning. If nervous semantic freshness is stale, the
layer may report readiness of the gate itself, but it must surface the stale
sidecar and the exact maintenance command instead of treating stale retrieval
as current truth.

Governance gates precede escalation. `host.governance-gates` in
`self-awareness capabilities` must expose concrete memory, resource, and mode
readiness instead of placeholder status fields, including the current memory
class, resource orchestrator status, mode effective state, launch policy, and
non-mutating owner policy.

Capability rows are a live matrix, not a status list. Each
`self-awareness capabilities.capabilities[]` row must carry the normalized
matrix contract: endpoint metadata where an HTTP route exists, latest artifact
schemas, freshness, history route, auth-safe read-only access policy, evidence
route counts, and owner-boundary fields proving the host layer does not mutate
`abyss-stack` or promote stack runtime by observation alone.

Observation events are a signal fabric, not loose JSON records. Each
`events[].fabric` row must preserve actor, entity, temporal bucket, spatial
owner surface, context links, redacted source query, evidence route, label
policy, and read-only/no-stack-mutation policy. Trace IDs, run IDs, thread IDs,
checkpoint IDs, and alert fingerprints belong in context links and correlation
keys, not in Loki labels.

Memory and space are a bounded overlay, not a direct claim that raw stack data
is machine truth. `self-awareness context` must expose `memory_space` with RAG
packet entries, maps/graph/memory/nervous freshness gates, Postgres/Neo4j/RAG
semantic backend state, evidence refs, and policy proving bounded retrieval,
no memory writeback, no KAG truth publication, and no stack mutation.
The same command must also expose `context_packet`, a bounded resident/operator
packet that carries compact correlation-context hashes, memory-space summary,
stack handoff action ordering, resident worker status, governance gates,
escalation gate state, read-only tools, evidence refs, redaction bounds, and
no-action/no-stack-mutation policy. It must not store raw private content, raw
context values, raw log bodies, prompts, messages, or tool payloads.
`self-awareness spatial-graph` must project the same overlay into graph nodes
and edges so reasoning can move across time, space, service, artifact, and
freshness-gate evidence without mutating stack-owned data services.

Warm-E2B is a resident cognitive worker, not a proof oracle. The
`warm-e2b.resident-cognitive-worker` capability must expose stack-owned serving
owner, endpoint health/latency, monitor timers, resource/thermal posture,
candidate queue summary, heartbeat eval scorecard, and a cognitive contract.
Each investigation must also expose `resident_cognitive_packet` with bounded
context sources, read-only tool inventory, hypothesis tests, contradiction
notes, evidence-cited summary, and resource/mode-gated E4B/Qwen escalation
state. The packet proves no model execution, stack mutation, memory writeback,
project-root write, or automatic action is authorized by that context.

The investigation graph is a checkpointed LangGraph-style loop, not a single
summary step. `self-awareness investigate` must preserve this node order:
`plan_queries`, `query_evidence`, `resident_context_packet`,
`reason_over_evidence`, `request_more_evidence`, `validate_evidence`,
`record_artifact`, `brief_reaction_candidate`, and
`write_semantic_conclusion`. The graph must expose resume metadata, replay
command, failure-recovery routes, evidence validation with zero failed checks,
artifact-record state, and a brief/reaction candidate that requires human
approval. `self-awareness replay` must replay the same node order, expose
conclusion diff, preserve stack handoff closure-readiness as a first-class
replay packet, and keep failure recovery non-mutating. No investigation or
replay node may execute remediation or mutate `abyss-stack`.

The response layer must preserve the causal lineage, not only a command
suggestion. Self-awareness alert candidates consumed by `abyss-machine
reactions --json` must carry `response_contract` with validated episode
metadata, source event, investigation/replay routes, risk, blast radius,
rollback, runbook candidate, approval gate, and evidence refs. `abyss-machine
responses --json` must copy that same contract into the response route and
keep `automatic_response=false`, `executes_commands=false`, and
`host_layer_mutates_stack=false`.

Open stack handoff blockers also enter that response layer as conservative
causal episodes, even when no runtime alert event exists. `episodes/latest.json`
must emit `stack_handoff_blocker` rows with `truth_level=handoff_candidate`,
affected stack service and handoff/action spatial nodes, closure blockers,
runbook/verifier evidence, and handoff-only policy. `alerts/latest.json` then
turns each such episode into a non-executing self-awareness candidate whose
response contract preserves the stack-owned requirement id and stack-owner
runbook route without executing the runbook or mutating `abyss-stack`.

AI multimodal readiness is concrete evidence, not a modality name list. The
`ai.multimodal.capability-map` capability must expose STT, embeddings, LLM text,
TTS, and NPU detail with source model refs, stack model roots as read-only
inputs, OpenVINO CPU/GPU/NPU device readiness, LLM profile readiness, TTS
profile/eval evidence, host-managed cache/runtime boundaries, and explicit
policy that capability presence is not an `abyss-stack` promotion decision.

LLM escalation readiness is gated route evidence, not permission to run a
larger model. The `llm.escalation.routes` capability must expose the
`gemma4.workhorse` pack/review/validate/preflight chain, prove default E4B
review is sourceful and non-executing, expose current resource/mode gate state
for model execution, and map both Qwen3.6 lazy-load routes with context,
cpuset, prefill/slot cache, request, and stop commands. A blocked workhorse
preflight may still be full route coverage when the block is explicit and the
review-only route remains valid.

## Open Stack-Owned Requirements

`abyss-machine self-awareness capabilities --json` writes open gaps to
`{{ABYSS_MACHINE_STATE}}/self-awareness/requirements/latest.json`.
`abyss-machine self-awareness requirements --json` exposes those blockers as a
first-class stack handoff with owner, expected shape, evidence refs, and
machine-checkable acceptance contracts.
`abyss-machine self-awareness requirement-probes --json` evaluates every
stack-owned handoff acceptance contract against current bounded read-only
evidence and records whether the requirement is still open or closed by the
current probe. The same artifact now carries one stack-owner runbook candidate
per handoff, including proposed stack work, acceptance verifier steps, risk,
blast radius, rollback, and evidence refs. These candidates are handoff-only:
they do not authorize or execute stack mutation from `abyss-machine`.
`abyss-machine self-awareness stack-closure-dossier --json` joins those
requirements and probes into the stack-owner closure packet. It preserves one
entry per stack-owned requirement with priority/order, current state, fulfilled
and missing checks, dependency edges, reverse unblocks edges, closure impact,
coverage impact, closure evidence needed, runbook candidate, acceptance
verifier commands, evidence/artifact refs, and explicit
no-execution/no-stack-mutation/no-secret policy. This is the route to use when
the question is what remains for `abyss-stack`, in what order, which blockers
unblock other blockers, which self-aware organs remain incomplete, and how
machine will verify closure.
The same coverage impact is also preserved in
`brief.stack_handoff_action_map`, `context.context_packet.sections.stack_handoff`,
`investigate` request/validation/conclusion state, and
`replay.stack_handoff_replay`. A stack blocker is therefore not only an open
route: it names the affected AI-OS organ, blocked self-awareness planes,
affected stack/machine surfaces, closure value, proof commands, and the
handoff-only/no-stack-mutation policy throughout the reasoning loop.
`abyss-machine self-awareness failure-matrix --json` turns those gaps and
machine-owned guard failures into an explicit coverage matrix.
`abyss-machine self-awareness cycle --json` proves the full host-owned loop
from probe through investigation, replay, brief, reactions/responses, and
export while keeping stack-owned blockers explicit.

`abyss-machine self-awareness export --json` is the portable proof bundle. It
preserves the compatibility `artifacts` map and also emits `artifact_list` plus
`manifest` so each latest artifact has a name, expected schema, current schema,
history path, size, SHA-256, evidence ref, and owner-boundary policy.

Each `stack_handoff[]` item carries an `acceptance_contract` and
`machine_closure_probe`. The contract tells the stack route what shape closes
the requirement, which read-only probe fields must exist, which redaction rules
must hold, which machine verifier commands must pass, and what must never
happen. A 2xx endpoint alone is not closure; closure requires current evidence
refs, a stack-owned bounded route, redaction/passive-read guarantees, and green
machine validators.

Each `requirement-probes.probes[]` row must preserve that same closure contract:
the row carries `stack_handoff: true`, `acceptance_contract`, `machine_closure_probe`,
`acceptance_verifiers`, `closure_semantics`, live current state, checks, evidence
refs, and a handoff-only stack-owner runbook candidate. Future stack work should
be able to pick up a probe row directly without guessing the missing verifier
shape.
Each row also carries `closure_readiness`: fulfilled checks, missing checks,
blocking check keys, dependency requirements, evidence still needed for closure,
verifier commands, safe next action, and no-stack-mutation policy. This packet
is owner-facing handoff guidance, not permission for `abyss-machine` to execute
the stack runbook.

`stack-closure-dossier/latest.json` is the normalized closure entrance when the
handoff spans multiple artifacts. Its `dependency_graph` orders open
requirements and reverse unblocks edges, its `stack_owner_handoff` carries the
verifier chain, and each entry must carry `closure_impact` so the stack owner
can see which remaining requirements are downstream of a blocker. Each entry
must also carry `coverage_impact` so the stack owner can see which
self-awareness planes and AI-OS organs are incomplete because of the blocker.
Each entry also carries a `compat_contract`, and the dossier exposes the same
contracts in top-level `compat_contracts`, so the stack owner gets the exact
route/export options, required fields, redaction/boundedness rules, dependency
order, post-close machine verifiers, and no-stack-mutation boundary without
manual joins.
Each entry must be complete enough for the stack owner to implement the missing
stack-owned route and then rerun the listed machine validators. The dossier is
still read-only: it never executes the runbook and never grants
`abyss-machine` ownership over stack configs, dashboards, datasources, traces,
database schemas, or LangGraph runtime APIs.

`export/latest.json` is the portable one-file handoff for this surface. In
addition to the artifact manifest, it carries top-level `requirements` and
`stack_handoff` sections with open stack requirement ids, closure blockers,
current state, runbook candidates, acceptance verifier steps, artifact refs, and
non-mutating policy. A stack owner should be able to start from that export
without manually joining `requirements/latest.json` and
`requirement-probes/latest.json`.
The same export preserves `closure_readiness` packets so stack-owner work can
start from one portable artifact with fulfilled checks, missing evidence,
dependency order, and verifier commands intact.

`brief/latest.json` is the operator-facing handoff entrance. Its
`stack_handoff_action_map` prioritizes the open stack-owned blockers and carries
the safe next action, closure blocker keys, current state, runbook candidates,
verifier commands, evidence refs, and no-stack-mutation/no-secret/no-action
policy directly in the semantic brief.

`investigate/latest.json` must carry that same `stack_handoff_action_map` through
the replayable loop, not leave it as a brief-only summary. The
`request_more_evidence` state emits one `stack_handoff_action` row for each open
action with closure blockers, runbook candidate, verifier commands, safe next
action, evidence refs, and handoff-only policy. `brief_reaction_candidate`, the
top-level investigation payload, and the semantic conclusion preserve the same
map and top action so replay can prove what stack-owned work remains before any
owner-routed mutation.

The same investigation, replay, and cycle artifacts must preserve
`stack_handoff_closure_readiness` as a separate contract. The packet keeps the
ordered next actions, fulfilled/missing checks, blocking keys, dependency
edges, verifier commands, safe next action, evidence refs, and no-execution
policy for every open stack-owned blocker. `replay.stack_handoff_replay` must
prove that `request_more_evidence`, `brief_reaction_candidate`, and
`write_semantic_conclusion` all kept that readiness packet before the E2E cycle
can claim covered status.

`cycle/latest.json` must also carry `from_zero_proof`: an artifact-level proof
map over the current run. Each proof step names the command, machine-owned
latest path, schema, hash, freshness metadata, evidence refs, and non-mutating
policy for the capability, signal-fabric, timeline, spatial graph, context,
episodes, alerts, warm-E2B, RAG/memory, investigation, replay, reaction,
response, and export path. Each chain obligation must point back to one or more
proof steps. A cycle cannot be `covered` if the proof map is missing, stale,
hashless, outside `{{ABYSS_MACHINE_STATE}}`, or mutation-capable.
The same cycle must carry `bridge_proof`, with explicit rows for heartbeat,
memory, mode, resource, process snapshot, container/process health, thermal
plan, cooling, typing events, typing validation, nervous brief, reactions, and
responses. These rows prove that the machine bridge organs named in the OS
Abyss objective are present as machine-owned readmodels before the cycle claims
full E2E coverage.

`timeline/latest.json` and `spatial-graph/latest.json` must also preserve the
same stack handoff as time/space evidence. `timeline.stack_handoff_time_space_overlay`
records one marker per open stack-owned blocker, with the observation bucket,
owner, affected service nodes, closure blockers, runbook, verifier commands,
safe next action, evidence refs, and non-mutating policy. `spatial-graph` turns
those markers into `stack_requirement`, `stack_handoff_action`, and
`stack_runbook_candidate` nodes plus edges to affected stack services, so open
stack potential is visible in the machine's spatial model instead of only in
operator-facing summaries.

`episodes/latest.json` consumes the same time-space overlay and emits one
`stack_handoff_blocker` causal episode per open stack-owned requirement. These
episodes have no source `event_ids` by default because they represent capability
handoff blockers, not runtime incidents. They preserve the marker id, affected
spatial nodes, closure blocker keys, stack-owner runbook candidate, verifier
commands, safe next action, evidence refs, and explicit
`root_cause_claim=false` / no-stack-mutation / no-execution policy. `alerts`
must keep those episodes flowing into reaction and response candidates with the
same response-contract depth as alert-backed episodes.

Current requirement families:

- read-only Grafana datasource inventory; Grafana health/version and inferred
  Prometheus/Loki/Alertmanager/Tempo datasource candidates are consumed as live
  evidence when readable, but they do not close the blocker without a
  stack-owned authoritative datasource inventory route or bounded export;
- Tempo or equivalent trace backend for structural span joins; Prometheus,
  Alloy, Loki readiness, bounded LogQL, and traceparent queryability are
  consumed as live evidence when readable, but they do not close the blocker
  without a trace backend ready/search route and span/log/metric join support;
- stack-owned read-only Postgres/Neo4j semantic inventory; route-api/RAG
  health, bounded OpenAPI, RAG collections/sources/agentic-graph shape,
  Postgres readiness, and Neo4j root metadata are consumed as live evidence
  when readable, but they do not close the blocker without Postgres
  schema/freshness and Neo4j label/relationship/freshness inventory;
- stack-owned `langchain-api` health/OpenAPI discovery plus `/run`,
  `/run/federated`, `/embeddings`, OVMS auth/provider metadata, and runtime
  request schema names are consumed as live bounded evidence when available;
  the remaining blocker is read-only LangGraph thread/checkpoint/trace
  inventory plus trace-backend coupling that can support replay and spatial
  correlation without storing prompts, messages, tool payload bodies, or raw
  trace payloads.

These are not host-layer failures while they are correctly represented as
requirements and runbook candidates. They remain blockers for claiming the
whole OS Abyss stack potential is fully exhausted.

```bash

abyss-machine self-awareness stack-closure-dossier --json
abyss-machine self-awareness failure-matrix --json
abyss-machine self-awareness cycle --json
```

Then run the host-layer gates:

```bash

```

## Non-Claims

- This layer does not own `abyss-stack`.
- This layer does not claim full distributed tracing unless the evidence proves
  propagation.
- This layer does not infer root cause as fact from time adjacency alone.
- This layer does not execute remediation.
